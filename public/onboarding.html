Overview // Control Plane

[ WHAT_OPERATOR_OS_IS ]

What Operator OS is

Operator OS is the control plane that connects public diagnostics and intake to governed execution. It coordinates service selection, agent workflows, marketplace fulfillment, security controls, telemetry, and the authenticated Operator Cockpit.

  • Public diagnostics and intake — guided service selection, registration, and structured lead capture.
  • Governed AI agents — scoped agent builders and advisory flows with explicit approval gates.
  • Workflow automation — intake-to-fulfillment routing with operator review checkpoints.
  • Marketplace fulfillment — module catalog, entitlement checks, and delivery evidence.
  • Security and governance controls — non-certifying governance metadata, mutation gates, and audit trails.
  • Telemetry and evidence — structured conversion and operational signals without storing sensitive free text.
  • Operator Cockpit — authenticated command surface for queues, status, and operator actions.

Activation // Sequence

[ HOW_ACTIVATION_WORKS ]

How activation works

  1. Explore services — review public diagnostics and choose the best-fit intake path.
  2. Submit intake or registration — provide contact details and context for operator review.
  3. Receive operator review — submissions are persisted and routed to the operator queue.
  4. Schedule a consultation when appropriate — use booking when you need a faster handoff.
  5. Receive activation guidance — operators respond with readiness updates and next steps.
Boundary // Public vs Operator

[ PUBLIC_PRIVATE_BOUNDARY ]

Public and operator-only boundary

Registration and intake do not automatically grant cockpit access. Public forms create reviewable records; operator-only systems remain protected behind authentication, session enforcement, and governance gates.

  • No autonomous account creation on the public surface.
  • No automatic approval or instant cockpit unlock.
  • Operator review is required before protected surfaces open.
