# Staging Readiness — Phase 1 System-Wide Enforcement

Manual checklist for promoting governed enforcement to `ttx-operator-shell-staging`. No automated deploy in this packet.

## Phase targets

```text
PHASE_0::COMPLETE
PHASE_1::SYSTEM_WIDE_CLOSED
PHASE_2A::CODEX_DX_FOUNDATION_COMPLETE
PHASE_2B::REFACTOR_PROPOSALS_READY
PHASE_3::NOT_STARTED
```

## Required secrets (staging)

Set via Wrangler — never commit values:

```bash
wrangler secret put GOVERNANCE_RECEIPT_SIGNING_KEY --env staging
wrangler secret put BEACON_SIGNING_KEY --env staging
wrangler secret put AUTH_SIGNING_KEY --env staging
wrangler secret put OPERATOR_PASSWORD_HASH --env staging
wrangler secret put OPERATOR_CALLSIGN --env staging
```

`AUTH_SIGNING_KEY` is JWT-only. Receipt and beacon HMAC must use domain-separated keys.

## Pre-deploy validation (local)

```bash
npm ci
npm run typecheck
npm run build
npm run codex:validate
npm test
```

Expected codex gate: `ungoverned_c2_c6_routes: 0`.

## Beacon v2 publish (before staging smoke)

```bash
BEACON_SIGNING_KEY="<staging-beacon-key>" node scripts/publish-beacon-v2.mjs
npm run build
```

Verify `msh-ops/beacon/releases/current.json` signature uses domain `MSHOPS::BEACON_RELEASE::V2`.

## Staging deploy (manual)

```bash
npm run build
wrangler deploy --env staging
```

Apply migration `v37-receipt-authority` on first deploy with `ReceiptAuthority` DO.

## Smoke procedures

### Governance health

```bash
curl -s -H "Authorization: Bearer <token>" https://<staging-host>/api/operator/status | jq .governance
```

Expect:

- `beaconStatus`: `verified_v2`
- `receiptAuthorityAvailable`: `true`
- `status`: `healthy` (or `degraded` only for codex warnings, not blocked)

### Signed proposal → approve → execute

1. `POST /api/governance/proposals` with current beacon/codex hashes
2. `POST /api/governance/proposals/{id}/approve` with matching `actionType` + `mutationPayload`
3. Execute governed route (e.g. activation campaign create) with `proposalId`, `approvalId`, `idempotencyKey`

### Receipt consumption

- Repeat execution with same `approvalId` but new idempotency key → `RECEIPT_CONSUMED`
- Repeat with same idempotency key → idempotent replay

### Organizer report

```bash
curl -s -H "Authorization: Bearer <token>" "https://<staging-host>/api/operator/agents/organizer/report?scan=1"
```

Expect `organizer.scan.completed` telemetry and KV key `organizer:v1:latest`.

### Disabled legacy routes (staging)

These must return `503` / `ROUTE_DISABLED_PENDING_GOVERNANCE`:

- `POST /api/governance/propose`
- `POST /api/governance/approve`
- `POST /api/ttx/local-scenarios/create`
